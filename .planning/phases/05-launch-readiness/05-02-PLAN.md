## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Launch approval → Production deploy | Unauthorized person initiates deployment |
| Gate verification → Sign-off | False positive verification claims (gate passes but didn't really) |
| Infrastructure → Public internet | Configuration error exposes secrets |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Spoofing | Launch approval sign-off | Mitigate | Signature field + timestamp required; archived in version control with git commit |
| T-05-02 | Tampering | Release gates checklist | Mitigate | Gates published in RELEASE_GATES.md (version controlled); executor verifies against original before sign-off |
| T-05-03 | Repudiation | Launch decision evidence | Mitigate | LAUNCH_SIGN_OFF.md template requires recorded evidence for each gate result; saved to git history |
| T-05-04 | Information Disclosure | Sign-off contains env details | Accept | Sign-off template references gate results (not secrets); actual credentials remain in Railway vault only |
| T-05-05 | Denial of Service | Rollback playbook missing | Mitigate | ROLLBACK_PLAYBOOK.md (05-03) documents rollback steps; sign-off references rollback criteria |
| T-05-06 | Elevation of Privilege | Solo operator approves all gates | Accept | For v1: Solo launch is acceptable; future launches add secondary approver to Approvals section if needed |

</threat_model>
