**2. API Rollback (Fastest Path: < 5 minutes):**

Target: Revert application code to last stable version without stopping database.

**Steps:**

1. **Assess Damage (T+0 to T+2):**
   - SSH into Railway or check logs: `railway logs --service {app-name}`
   - Look for error patterns: auth failures, payment failures, database connection errors?
   - Check health endpoint: `curl https://{railway-domain}/api/health`
   - If health endpoint returns 500, proceed with rollback

2. **Identify Last Stable Commit (T+2 to T+3):**
   - Go to: https://railway.app → Project → Service → Deployments
   - Find the last deployment marked with a green checkmark (successful)
   - Note the commit hash and date (should be previous deploy, not current failed one)
   - Alternative: `git log --oneline | head -10` to see last commits locally

3. **Trigger Rollback in Railway (T+3 to T+5):**
   - Method 1 (Fastest): Railway Dashboard
     - Click on last stable deployment
     - Click "Redeploy" button
     - Wait for Railway to rebuild and deploy (~2 minutes)
     - Monitor: Railway shows deployment progress in real-time
   
   - Method 2 (Via Git): If Railway UI not available
     - Identify current broken commit: `git log -1 --oneline`
     - Create revert: `git revert {broken-commit-hash} -m 1` (if merge commit)
     - Or: `git checkout {stable-commit-hash}` (backup if revert not possible)
     - Commit: `git commit -m "revert({phase}): rollback from broken deploy"`
     - Push: `git push origin main`
     - GitHub Actions triggers automatically; Railway deploys

4. **Verify Rollback (T+5 onwards):**
   - Health check: `curl https://{railway-domain}/api/health` (should return 200)
   - Check logs: Are ERROR entries gone?
   - Sentry: Has error rate dropped?
   - Smoke test: Can you login? (auth gate)
   - Smoke test: Can you create checkout? (payment gate)
   
5. **If Rollback Fails:**
   - If Railway deployment hangs: Cancel deployment, try Method 2 (Git revert)
   - If Git revert conflicts: Contact support or manually fix conflicts
   - If health endpoint still failing: Database may be affected → proceed to Database Rollback section

**3. Database Rollback (If Schema Migration Failed: < 15 minutes):**

Target: Revert database schema to last known good state without losing user data.

**Prerequisites:**
- Railway auto-backs up PostgreSQL daily
- Identify backup timestamp (when schema was last good)

**Steps:**

1. **Assess Migration Damage (T+0 to T+3):**
   - Check migrations status: `prisma migrate status` (in local dev environment connected to production DB — risky, use with caution)
   - Or: Check Sentry for database errors (e.g., "column not found", "table not found")
   - Expected errors on failed migration:
     - "ERROR: relation 'NewTable' does not exist"
     - "ERROR: column 'field_name' does not exist"
     - "ERROR: foreign key constraint failed"

2. **Stop the Application (T+3 to T+4):**
   - Railway dashboard: Service → click "Pause"
   - This prevents the app from trying queries against broken schema
   - Check that application is not accepting requests

3. **Request Database Restore from Railway (T+4 onwards):**
   - Email Railway support: support@railway.app
   - Subject: "URGENT: Database recovery request for {project-name}"
   - Content:
     ```
     Project: YT Multi-Publisher v1
     Environment: Production
     Issue: Schema migration failed, need restore to point-in-time
     Backup Time: [Last known good time, e.g., "2026-04-28 14:30:00 UTC"]
     
     Failed migration: [describe what was attempted]
     Error message: [paste error from logs]
     
     Please restore database to most recent backup before this time.
     ```
   - Railway responds with restore timeline (usually 5-30 minutes)

4. **Wait for Restore (T+4 to T+20):**
   - Railway staff will restore your PostgreSQL to the backup point
   - You'll be notified when complete
   - No action needed from you

5. **Resume Application (T+20 onwards):**
   - Railway dashboard: Service → click "Resume"
   - Application restarts and reconnects to database
   - Health check: `curl https://{railway-domain}/api/health`
   - Verify: Database queries work again

6. **Fix Migration Locally & Re-Deploy (T+20 onwards):**
   - On your machine: Fix the Prisma schema
   - Run: `prisma migrate resolve --rolled-back {migration-name}` (mark rollback as resolved)
   - Create new migration with corrected schema: `prisma migrate dev --name fix_schema`
   - Test locally first
   - Commit and push to main
   - Railway auto-deploys (but no schema changes this time, just fixed code)

**Important Notes:**
- Database restore takes 15-30 minutes minimum (contact Railway for exact timeline)
- Data written since backup is lost (acceptable for v1, document in incident report)
- Only use this if API rollback isn't sufficient
- Always have a recent backup before deploying migrations

**4. Worker Rollback (If Campaign/Payment Job Fails: < 10 minutes):**

Target: Pause background workers, revert job code, restart workers.

Current workers: campaign-processing, tiktok-upload, instagram-upload, integrated-worker (handles payments, campaigns, uploads)

**Steps:**

1. **Identify Failed Worker (T+0 to T+2):**
   - Check Sentry dashboard: Look for worker-specific errors
   - Check logs: Search for "worker" or "job" errors
   - Common failures:
     - TikTok upload timing out (worker stops responding)
     - Instagram OAuth refresh expired (worker can't authenticate)
     - Campaign status update stuck (database deadlock)

2. **Pause Worker (T+2 to T+3):**
   - Option 1 (Fastest): Environment variable kill switch
     - Railway dashboard: Variables → Set `SKIP_WORKERS=true`
     - This disables all worker initialization
     - Application continues serving UI/API, but no background jobs run
   
   - Option 2: Restart individual worker
     - If only one worker is broken (e.g., tiktok-upload):
     - Look at `.env.production`: Find the worker's corresponding service/startup code
     - Disable just that worker if possible (requires code change, slower)

3. **Revert Worker Code (T+3 to T+5):**
   - Follow API Rollback steps (Section 2)
   - This reverts the entire app, including worker code
   - Application comes back online

4. **Resume Workers (T+5 to T+8):**
   - Railway dashboard: Variables → Remove `SKIP_WORKERS=true` (or set to false)
   - Application restarts with workers enabled
   - Check logs: Workers initialize without errors
   - Monitor: Are jobs resuming?

5. **Verify Workers (T+8 onwards):**
   - Check for failed jobs in logs (how: grep for "JOB FAILED" in Railway logs)
   - Monitor Sentry: Has worker error rate dropped?
   - Smoke test: Upload campaign, verify job processes (may be delayed due to queue backlog)

**1. Severity Level Definitions:**

Define 3 severity tiers with automatic actions:

**SEVERITY 1 (CRITICAL) — Immediate Rollback Required**

- **Definition:** Service unavailable, data at risk, or payment system down
- **Examples:**
  - Health endpoint returns 500 (application crashed)
  - Database connection lost (no queries working)
  - Payment processing not working (webhooks not received, checkout fails)
  - Authentication system down (all users locked out)
  - Data corruption detected (schema mismatch, constraint violations)
- **Response SLA:** 5-minute first response, 15-minute resolution
- **Automatic Action:** Page on-call engineer immediately (you, Lucas)
- **First-Hour Steps:**
  - [ ] T+0: Assess damage severity (is it truly CRITICAL?)
  - [ ] T+1: Page on-call if not already aware
  - [ ] T+2: Trigger rollback (use ROLLBACK_PLAYBOOK.md)
  - [ ] T+5: Confirm rollback success (health check passes)
  - [ ] T+10: Send brief internal update (Slack/email)
  - [ ] T+30: Send user notification (if applicable)
  - [ ] T+60: Complete incident log (severity, duration, root cause)
- **Communication:** Urgent, immediate notification to all stakeholders
- **Rollback Threshold:** YES (automatically execute rollback)

---

**SEVERITY 2 (HIGH) — Monitor + Owner Notification**

- **Definition:** Degraded service, partial functionality broken, errors affecting subset of users
- **Examples:**
  - OAuth login works, but TikTok OAuth fails (users can't connect TikTok)
  - Payment checkout works, but webhook processing delayed (users paid but not credited yet)
  - Campaign publishing works for YouTube, fails for TikTok (multi-platform broken)
  - Sentry receiving errors, but < 5% of requests failing (acceptable error rate)
  - R2 media upload slow (> 10s), but still working
- **Response SLA:** 15-minute first response, 1-hour resolution
- **Automatic Action:** Log incident, notify owner (you) via ticket/Slack
- **First-Hour Steps:**
  - [ ] T+0: Log incident in tracking system (Sentry, Slack thread, etc.)
  - [ ] T+5: Assess scope (which users affected? all or subset?)
  - [ ] T+10: Decide: Rollback or patch in place?
  - [ ] T+15: If patch: Deploy fix; If rollback: Execute ROLLBACK_PLAYBOOK.md
  - [ ] T+30: Send owner notification (no public notification yet)
  - [ ] T+45: Monitor recovery (error rate dropping?)
  - [ ] T+60: If resolved, end incident; if not, escalate to SEVERITY 1
- **Communication:** Internal notification, escalate if not resolved in 1 hour
- **Rollback Threshold:** NO (monitor, attempt patch, escalate if patch fails)

---

**SEVERITY 3 (MEDIUM) — Log + Monitor (No Escalation)**

- **Definition:** Non-critical issues, system operational but with reduced functionality
- **Examples:**
  - Non-essential feature broken (e.g., analytics dashboard slow)
  - Third-party integration rate-limited (TikTok API quota hit)
  - Monitoring/Sentry quota exhausted (errors not being logged)
  - Performance degradation (response time 2-5 sec, still acceptable)
  - Configuration issue not affecting users (e.g., wrong feature flag)
- **Response SLA:** 1-hour first response, end-of-day resolution
- **Automatic Action:** Log in Sentry/monitoring system, email owner
- **First-Hour Steps:**
  - [ ] T+0: Catalog issue (what's broken, which users affected?)
  - [ ] T+30: Assess if can be fixed without rollback (usually yes for SEVERITY 3)
  - [ ] T+60: Plan fix (patch, config change, or defer to next phase)
  - [ ] Monitor: Is it getting worse? If yes, escalate to SEVERITY 2
- **Communication:** Internal-only, no user notification required
- **Rollback Threshold:** NO (unless escalated)

**3. First-Hour Response Workflow:**

Universal sequence for any incident:

```
T+0:00 → DETECTION (alert triggered, error spiked, user reports issue)
T+0:05 → TRIAGE (assess severity: CRITICAL / HIGH / MEDIUM)
T+0:10 → ESCALATION (if CRITICAL, page on-call; if HIGH, log ticket)
T+0:15 → ASSESSMENT (impact scope: all users? one feature? one region?)
T+0:20 → DECISION (rollback vs patch vs monitor?)
T+0:30 → ACTION (start rollback or deploy patch)
T+0:45 → VERIFICATION (did action resolve the issue?)
T+1:00 → COMMUNICATION (notify stakeholders of status)
```

**Detailed steps:**

**T+0:00 — Detection Phase:**
- Alert triggered by: Sentry error spike, health check failure, manual user report, or monitoring dashboard
- Log incident: Slack #incidents channel (if exists), Sentry incident, or local log file
- Severity assessment: CRITICAL if "service unavailable", HIGH if "partial", MEDIUM if "degraded"

**T+0:05 — Initial Triage:**
Use decision tree to confirm severity:
```
Is service completely down? (health endpoint returns 500, no user traffic possible)
  ├─ YES → CRITICAL, proceed to T+0:10
  ├─ NO → Is a critical gate failing? (AUTH, PAYMENT, DATABASE)
          ├─ YES → CRITICAL, proceed to T+0:10
          └─ NO → Is > 10% of requests failing?
                  ├─ YES → HIGH, proceed to T+0:10
                  └─ NO → MEDIUM, proceed to T+1:00 (monitoring phase)
```

**T+0:10 — Page On-Call (CRITICAL only):**
- If CRITICAL severity: Immediate notification (phone call, SMS, urgent Slack message)
- Message: "CRITICAL incident: {brief description}. Immediate response required."
- Include: Severity, incident type, link to logs/Sentry

**T+0:15 — Assess Impact Scope:**
- Which users affected? All? Subset?
- Which features broken? Auth? Payment? Campaign publishing?
- Which geographic regions? (If applicable — for v1, single region)
- Estimated user impact: "1000 users affected" vs "1 user affected"
- Time it started: When did the error spike first appear?

**T+0:20 — Make Go/No-Go Decision:**
```
CRITICAL Incident?
  ├─ Data at risk or payment broken?
  │   └─ YES → ROLLBACK immediately (no decision needed)
  └─ NO (code bug only) → Decide: Rollback vs Patch?
      ├─ If fix is simple & safe: Deploy patch (< 5 min)
      └─ If uncertain: ROLLBACK (faster than debugging)

HIGH Incident?
  ├─ Affecting all users?
  │   └─ YES → ROLLBACK or deploy patch (whichever is faster)
  └─ NO (affecting subset) → Patch if safe, otherwise monitor

MEDIUM Incident?
  └─ Monitor & defer to next business day (no immediate action)
```

**T+0:30 — Execute Action:**
- If ROLLBACK: Follow ROLLBACK_PLAYBOOK.md (API Rollback is < 5 min)
- If PATCH: Git commit → git push main → Railway auto-deploys (2-3 min)
- If MONITOR: Set alert threshold, keep owner on standby

**T+0:45 — Verification:**
After action:
- Health check: `curl https://{railway-domain}/api/health` (should 200)
- Error rate: Check Sentry, look for declining error count
- User impact: Is feature working again?
- If YES → proceed to T+1:00; If NO → escalate (possibly to SEVERITY 1)

**T+1:00 — Communication:**
See section 4 (Communication Templates) below
